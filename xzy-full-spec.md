# Xzy — Полная спецификация проекта

## 0. Ядро продукта (одна фраза)

Xzy — архитектурная прослойка (middleware) поверх Cursor/Claude Code. Она не пишет код сама, а даёт этим инструментам два новых свойства: **долгосрочную память проектных решений** и **аргументированный голос Senior-инженера**, который спорит на основе этой памяти, а не на основе мнения.

Xzy не решает проблему потери контекста внутри одной сессии (это территория самих платформ). Xzy решает потерю контекста **между сессиями и во времени** — почему команда полгода назад отказалась от Redis и почему сейчас снова наступает на те же грабли.

## 1. Система памяти

### 1.1 Структура записи (ADR — Architecture Decision Record)

Каждое решение хранится не как текст лога, а как структурированная запись с двумя типами полей:

**Поля для смыслового поиска** (объединяются в один текст и превращаются в embedding):
- `summary` — что решили и почему, коротко
- `context` — обстоятельства на момент решения
- `rejected_alternatives` — что рассматривали и отклонили

**Поля для жёсткой логики MCP-сервера** (не для поиска по смыслу, а для быстрой фильтрации):
- `tags`, `technologies.used/rejected` — точные ключевые слова для дешёвой проверки до вызова LLM
- `priority_level` — P1/P2/P3
- `override_condition` — при каком условии решение можно пересматривать
- `relations.supersedes/superseded_by` — связи с другими решениями

### 1.2 Как работает поиск и проверка конфликта (5 шагов)

1. **Захват** — система определяет по сообщению/коммиту, что это решение, а не рядовая правка.
2. **Дешёвый фильтр по тегам** — проверка, есть ли пересечение по ключевым словам/технологиям в базе (это почти бесплатно, отсекает большинство нерелевантных случаев).
3. **Семантический поиск** — если тегового совпадения нет напрямую, ищем по смыслу через embeddings top-5 похожих записей.
4. **Проверка конфликта через LLM** — только на найденных кандидатах, дорогая модель определяет: конфликт/повтор/развитие идеи.
5. **Запись результата** — новое решение сохраняется с меткой связи к найденным старым.

### 1.3 Экономика
Стоимость запроса не растёт линейно с возрастом проекта (всегда top-5 релевантных, а не вся база), в отличие от ручных docs-файлов, которые читаются целиком каждый раз и дорожают с ростом. Точка перелома в пользу Xzy — проекты от нескольких месяцев жизни и десятков решений.

### 1.4 Плюсы / минусы / решения

| Минус | Решение |
|---|---|
| Холодный старт (база пустая) | Разовое сканирование кода/конфигов при подключении, генерация стартовой карты проекта |
| Что считать "решением" — субъективно | Human-in-the-loop подтверждение в первые недели, фильтр дообучается на этой обратной связи |
| Память устаревает | Статус "актуально / под вопросом" + агент сам переспрашивает при конфликте |
| Приватность архитектурных данных | Шифрование записей, self-hosted опция для B2B |
| Ложные совпадения при поиске | Порог релевантности — при слабом совпадении лучше промолчать |

## 2. Спорящий Senior — полная механика

### 2.1 Принцип: видимый tool result, а не скрытая инъекция

Важно: агент **сам** решает вызвать инструмент памяти, увидев в задаче признаки архитектурного решения — это стандартная, документированная работа MCP. Xzy возвращает **честный, видимый результат вызова инструмента**, а не тайно подменяет системный промпт. Это принципиально: скрытая инъекция контекста через MCP — это ровно тот паттерн, из-за которого у AI-редакторов находят уязвимости и внедряют защиту вроде обязательного подтверждения человека на MCP-операции. Xzy должен быть прозрачным по построению, а не полагаться на обход этой защиты.

### 2.2 Три агента с разделением ответственности

**Агент-Безопасник (P1: безопасность/критический сбой)**
- Не LLM-рассуждение "на глаз", а детерминированный статический анализ (по типу существующих SAST-инструментов уровня Semgrep/Bandit): regex + AST-паттерны на известные уязвимости (eval, SQL-инъекции, хардкод секретов, OWASP-паттерны).
- Если сработал чеклист — это автоматически P1, без обсуждений.

**Агент-Память (P2: архитектура/поддержка)**
- Дёшево проверяет по тегам, есть ли пересечение с прошлыми ADR.
- Есть пересечение → зовёт LLM для содержательного сравнения → если реальный конфликт — P2.
- Нет пересечения в базе → автоматически падает в P3 (см. холодный старт ниже — это важный нюанс).

**Агент-Стилист (P3: минорное)**
- Всё, что не поймал Безопасник и не имеет истории в Памяти — простой autofix без долгого спора.

### 2.3 Логика приоритетов на практике

| Приоритет | Как определяется | Поведение агента |
|---|---|---|
| P1 | Статический анализ безопасности | Спорит жёстко, требует явного override с фиксацией риска |
| P2 | Совпадение с ADR в памяти + подтверждённый LLM конфликт | Высказывает позицию, ссылается на прошлое решение и `override_condition`, но соглашается при настойчивости юзера |
| P3 | Нет совпадений ни в чеклисте безопасности, ни в памяти | Молча предлагает autofix, не спорит |

**Важный нюанс холодного старта:** в первые дни жизни проекта база памяти пуста → P2 физически не может сработать (не с чем сравнивать) → все архитектурные решения по умолчанию скатятся в P3. Решение: стартовое сканирование кода (см. 1.4) должно сразу создавать базовые ADR по обнаруженным паттернам ("используется монолит", "выбран PostgreSQL"), чтобы P2 заработал не с нуля, а с первого дня.

### 2.4 Протокол несогласия и override

1. Агент спорит, ссылаясь на конкретный `override_condition` из ADR — не абстрактно, а с измеримым критерием ("пересмотр при нагрузке от 2000 RPS — у нас сейчас так?").
2. Если пользователь настаивает после аргументированного возражения — агент не блокирует работу намертво (иначе продукт снесут), а выполняет **протокол фиксации риска**: пишет код, но явно помечает в ответе, что решение принято вопреки истории, и почему.
3. В память уходит запись с меткой `CRITICAL_BYPASS` (для P1) или `OVERRIDDEN` (для P2), привязанная к конкретному ADR ID — это и есть новый ADR, который в будущем сам станет частью истории проекта.

**Как считать "юзер настаивает дважды":** счётчик привязывается не к тексту сообщения (юзер может переформулировать), а к паре (session ID + ADR ID, с которым идёт конфликт). Так система не "забывает" спор, даже если пользователь изменил формулировку запроса.

### 2.5 Тон и калибровка (чтобы не отключили через неделю)

- Критика — только уровня решения, не уровня человека: безличные конструкции ("этот подход приведёт к...") вместо оценки навыков ("ты не знаешь...").
- Опора на объективные инженерные принципы (SOLID, YAGNI, DRY, OWASP) — это переносит авторитет с "мнения ИИ" на признанный консенсус индустрии.
- Низкая температура генерации для агента-критика (ближе к точному, не "эмоциональному" стилю ответа) и жёсткий запрет в промпте на сарказм/риторические вопросы.
- В память попадает только сухой технический остаток решения — эмоциональный шум из диалога не сохраняется, чтобы не отравлять будущие ответы агента.

## 3. Архитектура (как всё соединяется технически)

```
                    ┌─────────────────────┐
                    │   MCP-сервер (ядро)  │
                    │  память + поиск +    │
                    │  проверка конфликтов │
                    │  + P1/P2/P3 логика   │
                    └──────────┬───────────┘
                               │
              ┌────────────────┴────────────────┐
              │                                  │
   ┌──────────▼──────────┐          ┌───────────▼───────────┐
   │   Hooks               │          │  Dashboard             │
   │  автоматическая        │          │  ручное подтверждение  │
   │  подсказка модели      │          │  решений, просмотр      │
   │  проверить память       │          │  истории, override-лог │
   │  (видимо, не скрыто)   │          │                         │
   └───────────────────────┘          └────────────────────────┘
```

**Принцип разделения:** толстая бизнес-логика (память, поиск, P1/P2/P3, проверка конфликтов) живёт в собственном бэкенде Xzy. MCP-сервер и hooks — тонкие, легко заменяемые "провода" к этому бэкенду. Так изменения в протоколах Cursor/Claude Code не ломают всю систему, а требуют правки только тонкого слоя интеграции.

**Claude Code — основная платформа MVP** (hooks и MCP там зрелые и документированные). Cursor — поддерживается через MCP как дополнительная, но не основная платформа для первой версии, с учётом того что там ниже уровень зрелости этого функционала и выше внимание к безопасности MCP-инъекций.

## 4. MVP — минимальный работающий сценарий

**Порядок реализации:**
1. MCP-сервер с базовой памятью (ADR-структура, дешёвый фильтр по тегам + семантический поиск)
2. Один hook в Claude Code, который подсказывает модели проверить память перед архитектурным решением
3. Простой лог/консольный вывод вместо полноценного dashboard (красивый UI — не День 1)
4. Human-in-the-loop подтверждение прямо в чате ("запомнить это? Да/Нет") вместо отдельного интерфейса
5. Только P1 (базовый чеклист безопасности) и P2 (сравнение с памятью) — P3 можно не выделять отдельно на старте

**Метрика успеха MVP:** не количество записанных решений, а количество случаев, когда пользователь подтвердил, что найденная связь была полезной.

## 5. Честные ограничения продукта

- Не решает потерю контекста внутри одной длинной сессии — это зона платформ (Cursor/Claude Code уже сами работают над этим через компрессию контекста и изоляцию подзадач).
- Люди с уже работающим ручным процессом (свои `.md`-файлы, дисциплина шаг-за-шагом) — самая тяжёлая аудитория для конверсии; не первая цель.
- Первые реальные клиенты: команды (не соло-разработчики) на средних/долгоживущих проектах, где ручной костыль либо ещё не появился, либо уже сломался под нагрузкой.

## 6. Открытые вопросы — финальный список перед стартом

1. Формат для первого демо — MCP+hooks внутри Claude Code, или всё же нужен минимальный отдельный интерфейс для первого показа не-разработчикам (инвесторам)?
2. Какая модель используется для дешёвого/дорогого этапов проверки конфликта — одна и та же LLM с разным промптом, или разные модели по цене?
3. Кто и как размечает начальный чеклист безопасности для Агента-Безопасника — берём готовый open-source SAST-инструмент или пишем свой список паттернов с нуля?
4. Порог релевантности для семантического поиска — какое стартовое значение, чтобы не спамить и не молчать?
5. Self-hosted/шифрование — делаем в MVP или откладываем до первого запроса от компании?
6. Как визуально/текстово должно выглядеть сообщение агента при P1-override — финальная формулировка предупреждения?
7. Кого из 4-5 опрошенных разработчиков зовём тестировать первую рабочую версию, и что именно считаем сигналом "это сработало" на их стороне?
